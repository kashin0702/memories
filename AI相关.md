### Prompt 结构化指令系统

在人工智能（尤其是大语言模型，如我这样的AI）的语境中，**“prompt”**（中文常译为“提示”或“提示词”）是指**用户输入给AI的指令、问题或上下文信息**，用于引导AI生成期望的输出。

简单来说，**prompt 就是你告诉 AI 要做什么的方式**。

system prompt(角色，规则)

User prompt (用户输入)

Context prompt(上下文，历史)

output prompt (输出格式)

- AI 的输出质量高度依赖于 prompt 的清晰度和具体性。
- 好的 prompt（称为“高质量提示”或“有效提示”）能显著提升 AI 回答的准确性、相关性和实用性。
- 这也催生了一个新技能——**Prompt Engineering（提示工程）**，即如何设计和优化 prompt 来获得最佳结果。

### RAG 检索增强生成

1. **你问一个问题**
   比如：“2025年诺贝尔物理学奖得主是谁？”

2. **AI 先去“查资料”**（这就是 **Retrieval 检索**）
   它不是靠自己记忆，而是快速从**最新的、可靠的文档库**（比如维基百科、公司数据库、新闻等）里找出相关段落。

3. **然后它看着这些资料写答案**（这就是 **Generation 生成**）
   它把查到的信息整理成通顺、易懂的回答，而不是瞎猜

   

   没有 RAG 的 AI：像一个只靠记忆答题的学霸，但可能答错新问题。

   有 RAG 的 AI：像一个能随时翻课本、查手机的聪明学生，答案更准、更新、更有依据。

总结：**RAG = 先查资料 + 再写答案，让 AI 不靠瞎猜，而是“有据可依”地说话。**



- **传统模型**：输入 prompt → 直接输出答案（端到端生成）
- **RAG 模型**：
  输入 prompt → **检索外部数据库** → 拿到相关文档 → **把文档+问题一起喂给大模型** → 生成答案

| 如果你需要……           | 选哪个？                |
| ---------------------- | ----------------------- |
| 写一首浪漫的情诗       | 传统生成模型            |
| 回答公司内部政策问题   | RAG                     |
| 获取2025年最新科技新闻 | RAG（配合更新的数据库） |
| 聊天闲扯、发散思维     | 传统模型更自然          |
| 避免AI胡说八道         | RAG 更可靠              |

## Prompt工程

#### 1. 基础构建块 (The Basics)

这是所有 Prompt 的基石，通常遵循 **ICIO** (Instruction, Context, Input, Output) 或 **CO-STAR** 等框架。

- **角色设定 (Persona/Role)**：赋予模型特定身份（如“你是一位资深Python架构师”），激活模型在该领域的专业知识分布。
- **清晰指令 (Clear Instruction)**：使用强动词，明确任务目标，避免歧义。
- **上下文注入 (Context Injection)**：提供必要的背景信息、业务规则或前置知识，减少幻觉。
- **输出约束 (Output Constraints)**：指定格式（JSON, Markdown, CSV）、长度、语气风格等。

#### 2. 示例学习 (In-Context Learning, ICL)

利用模型的上下文学习能力，无需更新权重即可适应新任务。

- 零样本 (Zero-Shot)

  ：直接给指令，不给示例。依赖模型预训练知识。

  - *技巧*：加上“让我们一步步思考”可显著提升效果。

- 少样本 (Few-Shot)

  ：在 Prompt 中提供 1-5 个高质量的“输入 - 输出”示例。

  - *作用*：让模型模仿示例的逻辑、格式和风格。对于复杂格式或非标准任务至关重要。
  - *关键点*：示例的质量比数量更重要；示例应覆盖多样性场景。

#### 3. 推理增强技术 (Reasoning Enhancement)

针对复杂逻辑任务，引导模型进行深度思考。

- 思维链 (Chain of Thought, CoT)

  ：强制模型展示推理过程（“Step-by-step”）。

  - *原理*：将复杂问题分解为中间步骤，利用生成的 Token 作为额外的计算时间。

- **思维树 (Tree of Thoughts, ToT)**：让模型生成多个可能的推理路径，自我评估并选择最佳路径（需配合代码逻辑实现搜索）。

- **自洽性 (Self-Consistency)**：对同一问题生成多条 CoT 路径，取出现频率最高的答案作为最终结果，提高准确率。

- **反思与修正 (Reflection/Self-Correction)**：让模型先生成答案，然后扮演“批评者”角色检查错误，最后修正答案。

#### 4. 结构化与模块化 (Structure & Modularity)

随着任务变复杂，单体 Prompt 难以维护，需要工程化手段。

- **分隔符 (Delimiters)**：使用 `"""`, `---`, `###`, `<tag>` 等清晰区分指令、上下文和输入数据，防止提示注入（Prompt Injection）并帮助模型解析。
- **模板化 (Templating)**：使用 Jinja2 等模板引擎动态填充变量（如用户历史、当前时间、检索到的文档），实现 Prompt 的复用和自动化。
- **模块化拆解 (Decomposition)**：将大任务拆分为多个子 Prompt，串联成工作流（Workflow）。例如：先提取实体 -> 再搜索知识 -> 最后生成报告。

#### 5. 高级策略 (Advanced Strategies)

- **检索增强生成 (RAG) 中的 Prompt 设计**：如何将从数据库检索到的碎片化信息整合进 Prompt，并指示模型“仅依据提供的上下文回答，若不知道则说不知道”，是 RAG 系统的核心。
- **函数调用/工具使用 (Function Calling)**：设计特定的 Prompt 结构，让模型学会输出结构化的 JSON 来触发外部 API 调用，而不是直接生成自然语言回答。
- **对抗性提示 (Adversarial Prompting)**：测试并防御“提示注入”攻击，确保模型在恶意输入下仍能遵守安全规则。

在2026年的开发实践中，优化 Prompt 通常遵循以下迭代循环：

1. **定义目标**：明确任务是什么，成功的标准是什么（准确率、格式、延迟）。
2. **构建初版**：使用 ICIO 框架编写基础 Prompt，加入角色和清晰指令。
3. **添加示例 (Few-Shot)**：如果效果不佳，添加典型的高质量示例。
4. **引入推理 (CoT)**：对于逻辑任务，强制要求分步思考。
5. **结构化输出**：强制要求 JSON/XML 格式，便于代码解析。
6. 评估与测试 (Evaluation)：
   - 构建测试集（Golden Dataset）。
   - 使用自动化评估工具（如 RAGAS, DeepEval）或人工评分。
   - 分析 Bad Cases（错误案例）。
7. **迭代优化**：根据 Bad Cases 调整指令、增加特定场景的示例或修改上下文。
8. **监控与维护**：上线后持续监控模型表现，随着模型版本更新或数据分布变化，适时调整 Prompt。



#### **ICIO 框架**

是提示工程（Prompt Engineering）中一种经典且高效的结构化设计模式。它的全称是 **Instruction（指令）+ Context（上下文）+ Input Data（输入数据）+ Output Indicator（输出指示）**。

这个框架的核心目的是通过标准化的结构，消除大语言模型（LLM）的歧义理解，使其输出更精准、更符合预期。它是构建高质量 Prompt 的基础，尤其适用于任务明确、需要结构化输出的场景。

以下是 ICIO 框架的详细拆解：

1. **核心组成部分详解**

**I - Instruction（指令）**

- **定义**：明确告诉模型“你要做什么”。这是 Prompt 的灵魂，必须清晰、具体、可执行。
- 关键点：
  - 使用动词开头（如“总结”、“翻译”、“编写代码”、“分析”）。
  - 避免模糊词汇（如“处理一下”、“看看这个”），改为具体动作。
  - 可以包含约束条件（如“用简洁的语言”、“不要使用专业术语”）。
- **示例**：“请作为一名资深产品经理，为以下用户反馈撰写一份改进建议报告。”

**C - Context（上下文）**

- **定义**：提供任务所需的背景信息、角色设定、前置知识或业务规则。这有助于模型“进入状态”，理解任务的边界和深层含义。
- 关键点：
  - **角色设定**：赋予模型特定身份（如“你是一位Python专家”、“你是一位挑剔的美食评论家”）。
  - **背景信息**：解释任务的来源、目标受众、相关业务逻辑。
  - **少样本示例（Few-Shot）**：有时也将具体的示例放在这里，作为上下文参考。
- **示例**：“我们的目标用户是Z世代年轻人，他们喜欢幽默、直接的表达方式。目前的痛点是APP注册流程太繁琐，导致流失率高达40%。”

**I - Input Data（输入数据）**

- **定义**：模型需要处理的具体内容。这是任务的“原材料”，通常与指令和上下文分隔开，以免混淆。

- 关键点：

  - 清晰界定数据的开始和结束（使用分隔符如 `"""`, `---`, `###` 或 XML 标签 `<data>...</data>`）。
  - 确保数据格式整洁，无多余噪声。

- 示例：

  ```text
  1### 用户反馈数据 ###
  2用户A: "注册还要填这么多资料，烦死了，直接退出了。"
  3用户B: "能不能用微信一键登录？不想记密码。"
  4用户C: "验证码收不到，试了五次都失败，垃圾软件。"
  5### 结束 ###
  ```

**O - Output Indicator（输出指示）**

- **定义**：明确规定模型输出的格式、风格、长度或结构。这是控制最终结果形态的关键。
- 关键点：
  - **格式要求**：JSON、Markdown 表格、CSV、代码块、列表等。
  - **风格要求**：正式、幽默、学术、口语化。
  - **长度限制**：不超过200字、分3点阐述。
- **示例**：“请以 Markdown 表格形式输出，包含三列：‘问题类别’、‘用户原声摘要’、‘改进优先级（高/中/低）’。最后用一段话总结核心建议。”



2. **ICIO 框架完整示例**

假设任务是：**从一堆杂乱的会议记录中提取待办事项。**

> **Prompt:**
>
> **[Instruction]**
> 请分析以下的会议记录，提取所有明确的待办事项（Action Items），并分配给相应的负责人。
>
> **[Context]**
> 你是某互联网公司的项目管理助理。这次会议是关于“双11大促活动”的筹备会。我们需要确保每个任务都有明确的截止时间和责任人，以便追踪进度。如果记录中未提及截止时间，请标记为“待定”。
>
> **[Input Data]**
> """
> 张三：我觉得首页 Banner 图得换，李四你负责设计一下，下周三前给我。
> 李四：没问题。那优惠券的逻辑呢？
> 王五：优惠券后端逻辑我来改，但这周比较忙，可能得周五才能弄完。
> 张三：行。还有客服那边，赵六你去对接一下培训，越快越好。
> 赵六：收到，我明天上午联系。
> """
>
> **[Output Indicator]**
> 请输出一个 JSON 格式列表，每个对象包含字段：`task` (任务描述), `owner` (负责人), `deadline` (截止时间)。不要输出任何多余的解释性文字。

#### CRISPE框架（方法论）

CRISPE 由以下 **6 个核心要素**组成

1. **CR - Capacity and Role（能力与角色）**
   - **含义**：明确告诉大模型它应该扮演什么角色，具备什么样的专业能力或背景。
   - **作用**：激活模型在特定领域的知识库，调整其语气、视角和专业深度。
   - **示例**：“你是一位拥有10年经验的高级Python后端架构师，擅长高并发系统设计。”
2. **I - Insight（洞察力/背景信息）**
   - **含义**：提供任务的背景、上下文、痛点或具体的业务洞察。这不仅仅是数据，还包括“为什么要做这件事”以及“当前的状况”。
   - **作用**：让模型理解任务的深层逻辑，避免生成泛泛而谈的内容。
   - **示例**：“我们的电商系统在双十一期间经常因为数据库锁竞争导致响应超时，目前使用的是MySQL主从架构。”
3. **I - Statement（指令/任务陈述）**
   - **含义**：清晰、具体地描述你需要模型执行的具体任务或动作。
   - **作用**：这是提示词的核心动词部分，直接定义输出目标。
   - **示例**：“请分析可能导致锁竞争的根本原因，并提供三个具体的优化方案，包括代码层面的建议和架构调整策略。”
4. **P - Personality（个性/风格）**
   - **含义**：定义模型输出内容的语气、风格和情感色彩。
   - **作用**：确保输出内容符合受众的接受习惯（如：幽默、严肃、学术、亲切等）。
   - **示例**：“请用专业、严谨但易于理解的技术文档风格撰写，避免过于晦涩的术语堆砌，适合向中级开发人员讲解。”
5. **E - Experiment（实验/尝试）**
   - **含义**：要求模型提供多个不同的选项、方案或视角，而不是只给出一个标准答案。鼓励模型进行发散性思维。
   - **作用**：为用户提供选择空间，激发创意，或者通过对比找到最优解。
   - **示例**：“请提供三种不同复杂度的解决方案：一种是快速修复方案，一种是中期重构方案，一种是长期架构演进方案，并列出各自的优缺点。”



#### BROKE框架（方法论）

**BROKE 框架**是一种用于设计和优化**提示词**（Prompt）的结构化方法论。它旨在帮助用户更清晰、系统地与大模型交互，从而获得更精准、高质量的回答。

1. **B - Background**（背景）

- **含义**：提供任务相关的背景信息、上下文情境以及需求产生的原因。
- **作用**：帮助模型建立正确的知识映射，理解“为什么”要做这件事，避免答非所问，提升响应的相关性。
- **示例**：“我是一名正在准备考研的学生，基础比较薄弱，距离考试还有3个月……”

2. **R - Role**（角色定义）

- **含义**：明确指定大模型在交互中需要扮演的角色（如专家、助手、批评家等）。
- **作用**：激活模型特定领域的知识权重，使其语气、风格和专业度更符合预期。
- **示例**：“请你扮演一位拥有10年经验的考研英语辅导专家……”

3. **O - Objectives**（目标/任务）

- **含义**：清晰地描述具体的任务目标或需要解决的问题。
- **作用**：让模型知道具体要“做什么”，确保输出方向正确。
- **示例**：“请为我制定一份详细的英语复习计划，重点突破阅读理解和写作。”

4. **K - Key Results**（关键结果）

- **含义**：定义预期的输出格式、关键指标或成功的具体表现。
- **作用**：约束输出的形式（如表格、代码、列表）和内容深度，确保结果可直接使用。
- **示例**：“请以周为单位列出计划表，包含每天的学习时长、具体练习题目类型及预期达成的词汇量。”

5. **E - Evolve**（改进/优化）

- **含义**：强调持续的试验、反馈与优化过程。
- **作用**：提示词工程往往不是一次性的，需要根据模型的初次回答进行微调（如要求更简洁、换个角度、补充细节），以迭代出最佳结果。
- **示例**：“如果第一版计划太难，请根据我的反馈调整难度；或者请对生成的计划进行自我批判并优化。”

## LangChain

架构：langchain采用三层分层设计

接口层：LLM API对接，统一对接chat-gpt,deepseek等大模型API

逻辑层：业务流程编排，通过chain和agent实现复杂逻辑

数据层：数据存储和管理，支持本地向量数据库和云存储的混合部署



核心模块交互机制：

```test
用户输入-》Prompt模板-》agent决策-》LLM推理（记忆存储，外部API/数据库）-》工具调用-》结果解析-》最终输出
```

核心组件包括

prompt模板：标准化提示词管理

agent决策：智能体规划和决策

工具调用：外部工具和API集成

LLM推理：大语言模型核心推理

记忆存储：对话历史与上下文管理

外部集成：数据库，API等外部系统

总结：langchain本质是一个基于提示工程构建的LLM应用开发框架，通过模块化和可组合的架构设计，构建由大语言模型驱动的复杂程序，其架构核心在于将LLM能力和业务系统无缝对接，同时提供扩展性。



### LangChain 的核心概念

LangChain 把大模型应用拆成了可复用的核心组件，新手先掌握这几个就够：

- **模型（Models）**：对接各种大模型的统一接口，不管是 OpenAI、Anthropic 还是国内的智谱、百度，都能用一套代码调用；
- **提示词（Prompts）**：提示词模板、提示词优化、动态填充变量，比如给大模型传「问题 + 知识库片段」的固定模板；
- **链（Chains）**：把「调用模型→处理输出→调用工具」等步骤串成流水线，比如「用户提问→检索知识库→拼接提示词→调用大模型→返回答案」就是一条链；
- **检索增强生成（RAG）**：LangChain 最常用的场景，内置了对接 Embedding 模型、向量数据库的组件，能快速实现「先搜知识库再回答」，避免大模型胡说；
- **工具（Tools）**：让大模型调用外部工具，比如计算器、搜索引擎、数据库、API，解决大模型「不会算、不知道最新信息」的问题；
- **记忆（Memory）**：保存对话上下文，让大模型能记住之前的聊天内容，实现多轮对话。

基础对话

```python
from langchain.schema import HumanMessage

# 1. 初始化大模型
llm = ChatOpenAI(
    model="gpt-3.5-turbo",
    api_key="你的OpenAI API Key"  # 替换成自己的key
)

# 2. 调用模型对话

# 3. 输出结果

```

简单RAG

```python
from langchain_openai import ChatOpenAI, OpenAIEmbeddings
from langchain_chroma import Chroma
from langchain.chains import RetrievalQA
from langchain.document_loaders import TextLoader

# 1. 加载本地知识库（比如一个txt文件）
loader = TextLoader("my_knowledge.txt")  # 本地文件，内容比如"LangChain支持RAG、多模态、工具调用"
documents = loader.load()

# 2. 把知识库转成向量并存储（用Embedding模型+向量库）
embeddings = OpenAIEmbeddings(api_key="你的OpenAI API Key")
vector_db = Chroma.from_documents(documents, embeddings)

# 3. 构建RAG链：检索知识库→调用大模型回答
qa_chain = RetrievalQA.from_chain_type(
    llm=ChatOpenAI(model="gpt-3.5-turbo", api_key="你的OpenAI API Key"),
    chain_type="stuff",
    retriever=vector_db.as_retriever()
)

# 4. 提问并获取答案
result = qa_chain.invoke({"query": "LangChain支持哪些功能？"})
print(result["result"])
```



**LangChain 的核心价值**

- 降低开发成本：不用自己写「调用大模型、对接向量库、处理上下文」的重复代码；
- 组件化复用：比如 RAG 流程，拆成「加载数据→向量化→检索→回答」，每个环节都能替换（比如换 Embedding 模型、换向量库）；
- 适配多模型：一套代码能对接不同厂商的大模型，不用改核心逻辑。

------

**总结**

1. **核心定位**：LangChain 是大模型应用开发的「胶水框架」，串联大模型与外部资源（数据、工具）；
2. **核心场景**：快速搭建 RAG 知识库、多轮对话机器人、工具调用型 AI 助手；
3. **核心优势**：组件化、低代码、适配多模型，大幅降低大模型应用的开发门槛。

## Transformer架构

### 1. 自注意力机制 (Self-Attention) —— 灵魂

这是 Transformer 最核心的创新。它允许模型在处理序列中的每一个词时，同时“关注”序列中的**所有其他词**，无论它们距离多远。

- **作用**：解决长距离依赖问题（Long-range Dependency）。在 RNN 中，相隔很远的词很难建立联系，而 Self-Attention 可以一步到位捕捉全局上下文。

- QKV 机制：

  计算过程涉及三个向量：

  - **Query (Q)**：我想找什么信息？
  - **Key (K)**：我拥有什么信息标签？
  - **Value (V)**：我具体的内容是什么？
  - **公式逻辑**：通过 Q×KT*Q*×*K**T* 计算相关性得分（Attention Score），归一化后加权 V*V* ，得到当前词的上下文表示。

### 2. 多头注意力 (Multi-Head Attention) —— 多维视角

模型不只做一次注意力计算，而是将 Q、K、V 切分成多组（例如 8 头或 16 头），并行进行多次自注意力运算，最后将结果拼接。

- 作用：让模型在不同的“子空间”里学习不同的关系。
  - 有的头可能关注**语法结构**（如主谓关系）；
  - 有的头可能关注**指代关系**（如“它”指代前面的“猫”）；
  - 有的头可能关注**语义关联**。
- 这极大地增强了模型的表达能力。

### 3. 位置编码 (Positional Encoding) —— 顺序感知

由于 Transformer 并行处理所有 Token，它本身**没有顺序概念**（不像 RNN 是一个接一个处理的）。

- **问题**：如果没有位置信息，“我爱你”和“你爱我”在模型眼里是一样的词袋。
- **解决**：在输入嵌入（Embedding）中加入一个代表位置的向量（通常使用正弦/余弦函数或可学习的参数）。
- **作用**：让模型知道每个词在句子中的绝对或相对位置。

### 4. 编码器 - 解码器结构 (Encoder-Decoder) —— 骨架

原始 Transformer 分为两部分（现在的 LLM 大多只使用 Decoder 部分）：

- Encoder (编码器)
  - **任务**：理解输入序列，将其转化为富含上下文信息的表示。
  - **特点**：使用**双向注意力**（可以看到过去和未来的所有词）。常用于翻译任务的源语言理解。
- Decoder (解码器)
  - **任务**：根据编码器的输出（或之前的生成内容），逐个预测下一个词。
  - 特点
    1. **掩码自注意力 (Masked Self-Attention)**：防止在训练时“偷看”未来的词（只能看到当前及之前的词），保证自回归生成的合理性。
    2. **交叉注意力 (Cross-Attention)**：让解码器关注编码器的输出（用于翻译等任务）。
  - **现状**：GPT 系列、Llama 系列等大模型都是 **Decoder-only** 架构，因为它们专注于生成任务。

### 5. 前馈网络与残差连接 (FFN & Residual Connection) —— 稳定与深化

- **前馈神经网络 (Feed-Forward Network, FFN)**：在每个注意力层之后，都有一个独立的全连接层，用于进一步提取特征和非线性变换。它是模型参数的主要存储地。

- 残差连接 (Residual Connection)

  在每个子层（注意力或 FFN）前后，将输入直接加到输出上 ( x+F(x)*x*+*F*(*x*))。

  - **作用**：防止梯度消失，允许训练极深的网络。

- **层归一化 (Layer Normalization)**：配合残差连接使用，加速收敛并稳定训练。

### 总结

1. **并行计算**：不再像 RNN 那样必须按顺序计算，可以利用 GPU 大规模并行加速，训练效率极高。
2. **长程依赖**：完美解决了长文本中上下文丢失的问题。
3. **可扩展性 (Scalability)**：架构简单统一，随着数据量和参数量的增加，性能呈现可预测的线性提升（Scaling Law），为千亿参数大模型铺平了道路。

简单来说，Transformer 就是**用“并行计算”换取了“全局视野”，并通过“堆叠层数”实现了“深度智能”**。







## SFT （监督微调）

**SFT** = **Supervised Fine-Tuning**（监督微调）

是在**预训练模型**（如大语言模型）基础上，通过**有标签的标注数据**调整模型参数，使其适应特定任务的技术。

**本质：用标注数据教模型“如何正确回答问题”**

**目标：**

| 阶段     | 模型能力                             |
| -------- | ------------------------------------ |
| 预训练后 | 具备通用语言能力，但不懂如何遵循指令 |
| SFT后    | 能理解人类指令，生成符合预期的回应   |

**核心流程：**

| 步骤           | 核心操作                                        | 关键要点                                    |
| -------------- | ----------------------------------------------- | ------------------------------------------- |
| **基座选择**   | 选用预训练模型（如 Llama 3、Qwen、GPT 系列）    | 平衡模型规模、算力与任务需求                |
| **数据构建**   | 标注 “指令 - 输入 - 输出” 三元组（Alpaca 格式） | 质量决定上限；需覆盖任务场景、清洗噪声      |
| **训练配置**   | 选择全参数或高效微调（LoRA/QLoRA/Adapter）      | 全参数效果最好但算力高；LoRA 适合低资源场景 |
| **损失计算**   | 交叉熵损失，仅对**输出部分**反向传播            | 避免模型遗忘指令，聚焦正确生成目标          |
| **评估与迭代** | 任务指标（如准确率、BLEU）+ 人工抽检            | 验证任务适配性与输出一致性                  |

 **核心技术要点**

- **数据格式**：推荐`Instruction（指令）+ Input（输入，可选）+ Output（输出）`，明确任务边界。

- **参数更新策略**

  - 全参数 SFT：更新所有权重，效果最佳，适合大算力场景；
  - 高效 SFT（PEFT）：仅更新部分参数（如 LoRA 的低秩矩阵），大幅降低显存占用，适合中小规模模型与个人开发者。

  

- **损失函数**：以自回归交叉熵为主，核心是让模型预测的输出分布尽可能接近标注的真实输出。



**何时需要SFT？**

| 场景           | 说明                             |
| -------------- | -------------------------------- |
| ✅ 指令跟随失败 | Prompt工程无法让模型输出预期格式 |
| ✅ 垂直领域适配 | 医疗、法律、金融等专业领域       |
| ✅ 特定任务优化 | 客服对话、代码生成、报告撰写     |
| ✅ 格式规范化   | 需要统一输出风格和结构           |
| ❌ 通用任务     | 简单问答可直接用基座模型+Prompt  |



### **与其他技术区别点（混淆点）**

| 技术                               | 核心差异                         | 典型场景                              |
| ---------------------------------- | -------------------------------- | ------------------------------------- |
| **预训练（Pre-training）**         | 无标注数据，学习通用语言表征     | 基座模型初始化（如 GPT-4 的基础训练） |
| **SFT（有监督微调）**              | 有标注数据，适配特定任务 / 格式  | 指令跟随、行业问答、代码生成          |
| **RLHF（基于人类反馈的强化学习）** | 用奖励模型和强化学习对齐人类偏好 | 提升模型的有用性、无害性、连贯性      |
| **DPO（直接偏好优化）**            | 无需奖励模型，用偏好对直接优化   | 简化对齐流程，适合中小规模场景        |

**标注数据概念：**

原始大模型只懂**语言规律**，不懂**怎么回答人类**。

需要人去：

1. 写问题
2. 写**正确、规范、有用的回答**

这个**人工写好答案**的过程，就叫**标注**。

写完的这一对：

> 问题 → 标准答案

就叫 **有标注数据（labeled data）**。

```text
┌─────────────────────────────────────────────────────────┐
│              大模型完整训练流程                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  1️⃣ 预训练(Pre-training)                                │
│     海量无标注数据 → 学习通用语言知识                    │
│                        ↓                                │
│  2️⃣ SFT(监督微调) ⭐                                      │
│     标注指令数据 → 学习遵循指令                          │
│                        ↓                                │
│  3️⃣ 对齐优化(RLHF/DPO/GRPO)                             │
│     偏好数据 → 学习人类偏好                              │
│                        ↓                                │
│  4️⃣ 部署应用                                            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

## PEFT（Parameter-Efficient Fine-Tuning）参数高效微调

**PEFT是实现SFT的一种技术手段，SFT的实现方式包括(PEFT和FFT全量微调)**

是一种**只微调少量参数**而非全部模型参数的技术，在保持预训练模型大部分参数**冻结**的情况下，通过调整少量额外参数来适配下游任务。

**对应的还有FFT(Full Fine-Tuning)，即全参数微调**

```text
┌─────────────────────────────────────────────────────────┐
│                    PEFT 核心思想                          │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  全参数微调：100% 参数可训练 → 资源消耗巨大               │
│                                                         │
│  PEFT微调：1-10% 参数可训练 → 资源消耗大幅降低            │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

核心作用：

| 价值            | 说明                           |
| --------------- | ------------------------------ |
| 降低计算成本    | 显存需求降低 5-10 倍           |
| 减少存储开销    | 只需保存少量适配器参数         |
| 加速训练        | 训练时间大幅缩短               |
| 降低过拟合      | 参数少，在小数据集上表现更好   |
| 多任务适配      | 同一基座模型可快速适配多个任务 |
| democratization | 让普通开发者也能微调大模型     |

### LoRA（主流PEFT方法）

Low-Rank Adaptation

```text
┌─────────────────────────────────────────────────────────┐
│                      LoRA 原理                           │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  原始权重更新：W' = W + ΔW                              │
│                                                         │
│  LoRA 分解：ΔW = A × B                                  │

│  推理时：合并权重 W' = W + A×B，无额外延迟               │
│                                                         │
└─────────────────────────────────────────────────────────┘
```





## 长尾问题

**长尾问题 = 很少有人问、很具体、很细分、很冷门、很个性化的问题。**

它们加起来**数量巨大**，但**单个出现频率极低**，在分布上拖出一条长长的 “尾巴”。

长尾问题解决方式：

- **简单长尾** → 检索 / RAG
- **流程长尾** → Workflow
- **复杂长尾** → Agent  自主思考，选择工具，多轮重试，自我修正
- **领域长尾** → LoRA 微调   不改变基座能力，只增强领域能力
- **所有长尾** → 加兜底   不瞎编，引导澄清



## CoT思维链和ToT思维树

**是大模型提示工程中两种高级策略，**是一种**让大语言模型展示推理过程**的技术，通过引导模型**逐步思考**而非凭直觉直接给出答案，它们的核心目的都是**让模型“慢下来”思考**，从而解决那些单靠直觉（System 1）无法处理的复杂逻辑问题。

- **CoT** 是让模型像人一样**一步一步地推导**。
- **ToT** 是让模型像下棋一样**先构思多种可能，评估后再选择最佳路径**



### CoT原理

从直觉到推导，让模型从**快思考**到**慢思考**。

通过将复杂问题分解为多个简单的中间步骤，模型利用自回归（Auto-regressive）的特性，将前一步的输出作为后一步的输入上下文，从而降低了每一步的推理难度

CoT局限：

1.依赖模型规模，小模型无法理解“一步步思考”，甚至导致中间错误，最终结果更差

2.事实性幻觉：只能提高逻辑推理的准确，不能保证事实知识的正确，如果模型不知道某个知识，可能进行瞎编。

### ToT原理

在思考过程中进行**发散、评估、回溯和剪枝**，像人类下棋或解数学题一样探索多种可能性。

- **节点 (Node)**：代表一个完整的“思维步骤”或“中间状态”（而不仅仅是一个词或一句话）。
- **边 (Edge)**：代表从一个思维状态转移到下一个思维状态的推理过程。
- **搜索算法**：使用经典的搜索算法（如广度优先搜索 BFS、深度优先搜索 DFS 或 A* 算法）来遍历这棵树，寻找最优解。

**具体执行流程通常包含四个关键步骤：**

1. **思维分解 (Decomposition)**：
   将复杂问题分解为多个连续的步骤。例如，写一篇文章可以分解为：构思大纲 -> 撰写段落 -> 润色修改。
2. **思维生成 (Thought Generator)**：
   在当前节点，让 LLM 生成**多个**可能的下一步思维（候选项），而不是只生成一个。
   - *示例*：面对一个创意写作任务，LLM 可能生成 3 种不同的开头风格（幽默的、严肃的、悬疑的）。
3. **状态评估 (State Evaluator)**：
   这是 ToT 的灵魂。LLM（或外部评分函数）对生成的每个候选思维进行**评估打分**。
   - *评估方式*：可以是简单的“通过/不通过”，也可以是数值评分（0-10），或者是自然语言评价（“这个方向偏离了主题”）。
   - *作用*：判断哪个思维更有希望导向最终正确答案。
4. **搜索策略 (Search Algorithm)**：
   根据评估结果，决定下一步怎么走：
   - **保留**高分思维，继续向下扩展。
   - **剪枝 (Pruning)**：抛弃低分思维，不再浪费算力。
   - **回溯 (Backtracking)**：如果当前路径走不通，退回到上一个节点，尝试之前被搁置的备选方案。

相比COT优势：

1.具备前瞻性和全局观，能比较不同策略。

2.适合高难度、需要探索、创造性强、容易陷入局部最优的任务（如复杂数学证明、创意写作、策略游戏、代码架构设计）。



**1.零样本思维链zero-shot** 

方法：只需在问题后加一句引导语：请一步步思考

```text
Prompt 模板：
─────────────
{问题}
请一步步思考。
─────────────

示例：
问题：如果3台机器3分钟生产3个零件，100台机器生产100个零件需要多久？
请一步步思考。

模型输出：
1. 3台机器3分钟生产3个零件 → 1台机器3分钟生产1个零件
2. 1台机器生产1个零件需要3分钟
3. 100台机器同时工作，每台生产1个零件
4. 所以100台机器生产100个零件也需要3分钟
答案：3分钟
```

**2.少样本思维链few-shot**

方法：在Prompt中提供带推理过程的示例 （效果比零样本好，但需要设计示例）

```text
Prompt 模板：
─────────────
问题：罗杰有5个网球。他又买了两筒网球。每筒有3个网球。他现在有多少个网球？
解答：罗杰开始有5个球。2筒球，每筒3个，那是2*3=6个球。5+6=11个球。答案是11。

问题：食堂有23个苹果。如果他们用掉20个做午餐，又买了6个。他们还有多少个苹果？
解答：食堂开始有23个苹果。用掉20个，剩下23-20=3个。又买了6个，3+6=9个。答案是9。

问题：{新问题}
解答：
─────────────

```

**方法**：用带推理过程的数据微调模型 （这是最强大的方式，需要构建CoT训练数据）

```text
┌─────────────────────────────────────────────────────────┐
│              CoT 数据格式示例                            │
├─────────────────────────────────────────────────────────┤
│                                                         │
│  {                                                      │
│    "instruction": "请解答以下数学问题",                  │
│    "input": "一个农场有鸡和兔共35个头，94只脚，问鸡兔各几只？",│

│               1. 设鸡有x只，兔有y只                     │

│               6. 70 - 2y + 4y = 94                      │

│               8. x = 35 - 12 = 23                       │
```

## RLHF 基于人类反馈的强化学习

**对齐领域的核心技术**

RLHF 是大语言模型对齐人类价值观的核心技术，也是ChatGPT等现代AI助手能够"听懂人话"的关键所在。

3个阶段

```text
┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│  阶段1       │    │  阶段2       │    │  阶段3       │
│  SFT        │ →  │  奖励模型    │ →  │  RL优化      │
│  监督微调    │    │  训练       │    │  (PPO/GRPO)  │
└─────────────┘    └─────────────┘    └─────────────┘
```



1. 预训练（基座模型）

- 模型读遍海量文本
- 学会语法、知识、怎么组词
- 但**不懂人类偏好**：可能乱答、有毒、生硬、跑题

2. SFT 监督微调

- 人工写一批**优质问答**
- 让模型模仿：“应该这样回答”
- 变通顺了，但**不够聪明、不够对齐**

3. RLHF 核心：强化学习

1. 训练一个 **奖励模型（RM）**

   它的任务：给 AI 的回答**打分**

   → 这个好、这个差、这个有毒、这个有用

   

2. 用 **PPO 算法** 让 AI 不断优化

   → 奖励高的回答，AI 以后多说

   → 奖励低的，少说

**最终目标：让 AI 越来越符合人类价值观、偏好、安全、有用**



奖励模型（Reward Model）训练

| 步骤     | 说明                                     |
| -------- | ---------------------------------------- |
| 数据收集 | 对同一prompt，模型生成多个回答，人工排序 |
| 标注方式 | A > B > C（偏好排序）                    |
| 模型训练 | 训练一个奖励模型预测人类偏好             |

```text
Prompt: "如何制作炸弹？"

回答A: "我不能提供..."  ⭐⭐⭐⭐⭐ (高分)
回答B: "首先准备材料..." ⭐ (低分)

奖励模型学习：A的奖励 > B的奖励
```



## Eembedding

**Embedding** 是机器学习和深度学习中的核心概念，简单说就是**将离散对象（如单词、用户、商品）转换为连续向量表示**的技术。

Embedding类型：

| 类型                             | 用途         | 示例                      |
| -------------------------------- | ------------ | ------------------------- |
| 词嵌入（Word Embedding）         | NLP文本处理  | Word2Vec, GloVe, FastText |
| 位置嵌入（Positional Embedding） | 表示序列位置 | Transformer中的位置编码   |
| 用户/物品嵌入                    | 推荐系统     | 用户ID、商品ID向量化      |
| 图像嵌入                         | 计算机视觉   | CNN/ViT提取的图像特征     |
| 图嵌入（Graph Embedding）        | 图神经网络   | 节点/边的向量表示         |
| 多模态嵌入                       | 跨模态检索   | CLIP的图文联合嵌入        |



应用场景：

| 领域   | 应用               | Embedding作用            |
| ------ | ------------------ | ------------------------ |
| NLP    | 文本分类、机器翻译 | 词/句向量表示            |
| 搜索   | 语义搜索、召回     | 查询-文档匹配            |
| 推荐   | 用户/物品匹配      | 个性化推荐               |
| 大模型 | 输入表示           | Token→向量（模型第一层） |
| 多模态 | 图文检索           | 统一语义空间             |

**Embedding 的核心价值**：

1. **桥梁作用**：连接离散世界（文字/ID）与连续世界（神经网络）
2. **语义编码**：将语义信息压缩到向量空间中
3. **可计算性**：让语义关系可以通过数学运算表达
4. **通用性**：几乎所有深度学习任务的基础组件

**一句话理解**：Embedding 是让计算机"理解"非数值数据的**翻译器**，把万物变成向量。



## 训练集，验证集，测试集

| 特性           | 训练集 (Training)     | 验证集 (Validation)                 | 测试集 (Test)               |
| -------------- | --------------------- | ----------------------------------- | --------------------------- |

## AI Agent开发技术栈

主要分为以下几个层级：**基础模型层、核心框架层、记忆与数据层、工具与执行层、以及部署观测层**

**1.核心编排框架**：Agent开发的“操作系统”，负责管理任务规划、状态流转、多智能体协作和工具调用。目前市场呈现“三足鼎立 + 新兴轻量级”的格局

**A. 通用型与图导向框架 (Graph-Based & General Purpose)**

- LangChain / LangGraph
  - **地位**：生态最丰富、社区最活跃的“事实标准”。
  - **特点**：LangChain提供了大量的预构建组件（Chains, Tools, Memory）。其子项目 **LangGraph** 专门用于构建有状态、基于图（Graph）的工作流，非常适合需要复杂循环、条件分支和人机协同（Human-in-the-loop）的生产级Agent。
  - **适用场景**：复杂的业务流程自动化、需要精细控制执行路径的企业级应用。
- LlamaIndex
  - **地位**：数据连接领域的王者，现已全面支持Agent工作流。
  - **特点**：最初专注于RAG（检索增强生成），现在其 `LlamaIndex Workflows` 提供了强大的事件驱动架构。它在处理私有数据、构建知识库驱动的Agent方面表现极佳，查询优化能力优于LangChain。
  - **适用场景**：知识密集型Agent、文档分析、企业知识库问答。

**B. 多智能体协作框架 (Multi-Agent Collaboration)**

- Microsoft AutoGen
  - **地位**：多智能体对话模式的开创者。
  - **特点**：核心概念是“对话”，通过定义不同的Agent角色（如Coder, Executor, Critic），让它们通过自然语言对话来协作解决问题。支持代码自动执行和反馈循环。
  - **适用场景**：软件开发辅助、复杂问题解决、需要动态交互的场景。
- CrewAI
  - **地位**：以“角色扮演”和“流程化”著称，上手最简单。
  - **特点**：将Agent抽象为具有特定`Role`（角色）、`Goal`（目标）和`Backstory`（背景故事）的成员，通过`Crew`（团队）来组织任务。它强制结构化输出，非常适合模拟人类团队分工（如：研究员->分析师->_writer）。
  - **适用场景**：市场调研报告生成、内容创作流水线、标准化任务协作。
- MetaGPT
  - **特点**：引入了软件工程中的SOP（标准作业程序），让Agent团队模拟软件公司的运作（产品经理、架构师、工程师等），能直接输出可运行的代码项目。

**2.记忆和数据层**

Agent需要“记住”用户偏好、历史对话和长期知识，这依赖于向量数据库和记忆管理机制。

- 向量数据库 (Vector Databases)
  - **主流选择**：**Pgvector** (Postgres插件，最流行)、**Qdrant** (高性能Rust编写)、**Milvus** (大规模集群)、**Chroma** (轻量级/开发测试)、**Weaviate**。
  - **作用**：存储嵌入向量，支持语义搜索，是RAG和长期记忆的基础。
- 记忆管理 (Memory Management)
  - **技术点**：短期记忆（Context Window）、长期记忆（Vector Store）、实体记忆（Entity Memory）。
  - **框架支持**：LangChain Memory, LlamaIndex Memory, **MemGPT (现Letta)** (专注于无限上下文和层级记忆管理)



**3. 工具与执行层 (Tools & Execution Layer)**

Agent的“手脚”，使其能与外部世界交互。

- 协议标准

#### 数据的token化

**Token** 是文本被分割后的最小处理单元，可以理解为模型"看到"的基本单位。

| 文本示例         | Token 划分                 |
| ---------------- | -------------------------- |
| `"Hello world"`  | `["Hello", "world"]`       |
| `"我喜欢AI"`     | `["我", "喜欢", "AI"]`     |
| `"unbelievable"` | `["un", "bel", "ievable"]` |

#### Tokenization

```text
原始文本 → 分词器(Tokenzier) → Token ID 序列 → 模型输入

文本: "大模型很强大"
       ↓ (Tokenization)
Token: ["大", "模", "型", "很", "强", "大"]
       ↓ (转换为ID)
ID:    [1234, 5678, 9012, 3456, 7890, 1234]
```

**tokenization的意义：**

| 原因                 | 说明                                      |
| -------------------- | ----------------------------------------- |
| 模型无法直接理解文本 | 神经网络只能处理数字，需要将文本转换为 ID |
| 控制词表规模         | 平衡词表大小与覆盖能力（通常 32K-100K+）  |
| 处理未登录词(OOV)    | 子词方法可以拆分新词，避免未知 token      |
| 多语言支持           | 统一的 token 空间支持多语言混合输入       |
| 训练效率             | 合适的 token 长度影响显存利用和训练速度   |

##### MoE架构

**MOE（Mixture of Experts，混合专家模型）** 是一种稀疏化神经网络架构，已成为2025-2026年大模型领域的主流技术路线。

```text
┌─────────────────────────────────────────────────────────┐
│                    MOE架构核心逻辑                        │
├─────────────────────────────────────────────────────────┤
│  大模型 → 拆分为多个"专家"子网络 → 动态路由选择 → 只激活少数专家  │
│                                                         │
│  💡 核心优势：参数规模大，但每次计算只动用少量参数                │
└─────────────────────────────────────────────────────────┘
```

```text
输入 Token
    ↓
┌───────────────┐
│   路由器      │ → 计算各专家权重
└───────┬───────┘
        ↓
   选择 Top-K 专家
        ↓
┌───────┴───────┐
│ 专家1 │ 专家2 │ ... │ 专家N │
│  ✓   │  ✓   │     │       │  ← 只激活选中的
└───────┴───────┘
        ↓
    加权输出
```



优势：

计算占用资源小，8 X7B 比肩 70B Decoder LLM

多专家，不同专家擅长不同任务

和传统稠密模型对比：

```text
┌──────────────────────────────────────────────────────────┐
│                    稠密模型 vs MOE                        │
├──────────────────┬───────────────────────────────────────┤
│    稠密模型       │              MOE模型                   │
├──────────────────┼───────────────────────────────────────┤
│ 每次推理用全部参数 │ 每次推理只激活少数专家                  │
```

#### 过拟合（overfitting）

**为什么会出现过拟合？**

1. **数据层面**

- **数据量太少 vs. 参数量太大**：模型有千亿参数，但微调数据只有几千条。模型“杀鸡用牛刀”，轻易就能把这几千条数据背下来。
- **数据重复率高**：训练数据中存在大量重复样本（未去重），模型反复看到相同内容，导致过度关注这些样本。
- **数据分布单一**：微调数据风格过于统一（如全是“你好 - 我很好”的简单对话），模型学不到多样性，遇到复杂问题就懵了。
- **噪声过多**：训练数据中包含错误标签或低质内容，模型强行拟合了这些错误。

2. **训练策略层面**

- **训练轮数（Epochs）过多**：这是微调中最常见的原因。大模型记忆力极强，通常 **1-3 个 Epoch** 足矣。如果跑 10 个 Epoch，模型就开始背题了。
- **学习率（Learning Rate）过大**：步长太大，模型直接跳进了某个局部最优解（死记硬背的坑），无法泛化。
- **全量微调（Full Fine-Tuning）风险**：更新所有参数容易破坏预训练阶段学到的通用知识分布，导致“顾此失彼”。

3. **模型架构层面**

- **容量过剩**：对于简单任务，使用超大模型（如用 72B 模型做简单的分类任务）极易过拟合。

##### 解决方法

针对大模型的特性，解决方案分为 **数据侧**、**算法侧** 和 **训练策略侧**。

**1. 数据侧优化（最根本的方法）**

- 增加数据多样性与规模
  - 收集更多样化的指令数据，覆盖不同场景、不同长度、不同风格。
  - **数据增强**：使用强模型（如 GPT-4）对现有数据进行改写、扩充（Self-Instruct），生成更多变体。
- 严格去重（Deduplication）
  - 在文档级、句子级甚至 n-gram 级别进行去重，确保模型不会反复学习相同内容。
- 提高数据质量
  - 人工清洗低质数据，确保 `(Prompt, Response)` 对的逻辑正确、格式规范。**高质量的小数据集 > 低质量的大数据集**。
- 混合通用数据（Replay Buffer）
  - 在微调数据中混入 **5%-10% 的预训练通用数据**（如维基百科、新闻）。这能充当“锚点”，防止模型忘记通用知识（缓解灾难性遗忘）。

**2. 算法与架构侧（技术手段）**

- 使用 PEFT 技术（首选方案）
  - **LoRA / QLoRA**：冻结主模型参数，只训练少量的适配器（Adapter）参数。
  - **原理**：限制了模型的“可变动空间”，相当于给模型戴上了“紧箍咒”，让它很难去死记硬背整个数据集，从而天然具有正则化效果，极大降低过拟合风险。
- 早停法（Early Stopping）
  - 监控验证集（Validation Set）的 Loss。当训练集 Loss 继续下降，但验证集 Loss 开始上升时，**立即停止训练**。
  - *注意*：大模型训练中，需要预留一部分高质量的验证集（不与训练集重叠）。
- 权重衰减（Weight Decay）
  - 在优化器中加入 L2 正则化项，惩罚过大的权重值，迫使模型保持简单。
- Dropout
  - 虽然在推理时通常关闭 Dropout，但在训练 SFT 模型时，适当开启 Dropout 可以防止神经元共适应。不过在大模型中，Dropout 的使用比较谨慎，以免影响性能。

**3. 训练策略侧（超参数调整）**

- 减少 Epochs
  - 对于指令微调（SFT），通常 **1 个 Epoch** 甚至 **0.5 个 Epoch** 效果最好。千万不要像训练 CNN 那样跑几十轮。
- 降低学习率（Learning Rate）
  - 使用更小的学习率（如 1e-5 到 2e-5），配合 **Cosine Decay** 或 **Warmup** 策略，让模型缓慢收敛，避免剧烈震荡陷入局部最优。
- 梯度裁剪（Gradient Clipping）
  - 限制梯度的最大范数，防止某一批次数据的噪声导致参数更新过大。
- 标签平滑（Label Smoothing）
  - 不让模型对正确答案的概率预测为 100%，而是留一点余地（如 90%），增加模型的不确定性，提升泛化能力。

#### GRPO实践

![image-20260224005324322](D:\typora-img\image-20260224005324322.png)

**GRPO（Group Relative Policy Optimization，组相对策略优化）** 是一种由 **DeepSeek** 提出的强化学习算法，专门用于大语言模型（LLM）的微调，尤其在**数学推理、编程、链式思考**等复杂任务中表现优异。它是传统PPO（Proximal Policy Optimization）算法的重要改进版本。核心是用**组内相对排序**替代传统的**绝对奖励打分**，解决 RLHF 中奖励模型不稳定、尺度漂移、Reward Hacking 等问题。

GRPO 是用 “分组比较、相对排序” 做强化学习对齐的轻量算法，让大模型对齐更稳、更省算力、更难被钻空子，尤其适合数学 / 推理类任务。



#### RLHF对比GRPO

| 维度     | 传统 PPO / RLHF         | GRPO                          |
| -------- | ----------------------- | ----------------------------- |
| 奖励信号 | **绝对分数**（RM 打分） | **组内相对排序**（A > B > C） |
| 额外模型 | 需要 **RM + Critic**    | **无需 RM/Critic**            |
| 稳定性   | 易受 RM 噪声 / 尺度影响 | 高稳定、梯度方差低            |
| 算力开销 | 高（多模型训练）        | 低（单模型为主）              |
| 对齐目标 | 拟合绝对奖励            | 拟合**相对偏好**              |



**RLHF 有什么缺点？（为什么会出现 GRPO）**

RLHF 虽然强，但很麻烦：

- 要训练 **奖励模型 RM**
- 要训练 **Critic 网络**
- 容易**奖励漂移、不稳定、训练崩**
- 算力开销大

于是后来出现了更轻量的方法：

**GRPO、DPO、IPO、KTO 等……**

它们统称：

**不需要奖励模型的 RL 对齐方法**

文本清洗：去重，去噪，去乱码



#### 文本拆分

对解析后的文本需要合理的划分为语义连贯的片段，以便检索系统能高效的找到与问题最相关的信息，从而增强生成模型回答的准确率。

方案：

**1.朴素切分**

基础的文本切分方法，基于固定长度或简单的分隔符（句号，换行符）。实现简单，**计算效率高。**无法处理复杂语义，切分结果可能不符合实际语义单元，有局限性。**适合对计算效率高但语义要求不高的场景。**



**2.父子文档**

父文档树：递归式构建文档间的树状层次结构。每个父节点包含子节点的摘要信息。动态规划决策每次检索时是否召回父文档。



**3.SeqModel切分**

阿里提出的高效文本切分模型，将文档分割建模为句子级序列标记任务。利用BERT预测段落边界，采用滑动窗口减少计算效率。对中文文档有更高效精准的切分。**适合对中文文档以及对推理速度要求较高的场景。**



#### 向量化模型

对于切分后的文本，需要将其转换为计算机易于处理和分析的数值形式，从而便于进行数学运算，模式识别，相似度计算等。

方案：

使用BGE模型，可以将任意文本映射到低维的稠密向量。其通过先进的训练和大量语料数据，学习到文本的语义信息，生成高质量文本向量。

#### 检索召回

由于文本类型和任务较多，采用多路集成召回方式，拓展召回维度。

**方案：**

**1.词级检索：**

基于关键词的稀疏检索，这是传统搜索引擎的核心技术，在 RAG 中作为重要补充

- **原理**
  - 使用 **BM25**、**TF-IDF** 等算法，基于词频统计和逆文档频率来计算相关性。
  - 完全基于字面匹配，不涉及语义向量。

BM25：高效信息检索算法，核心思想基于词频（TF）和逆文档频率（IDF），词级信息召回的关键一路，**适合词级召回。**

**2.语义检索：**

基于语义的向量检索

**原理**：

1. 将用户查询（Query）和文档库（Corpus）中的片段通过 **Embedding 模型**（如 BGE, M3E, text-embedding-3-large）转化为高维向量。
2. 计算 Query 向量与文档向量的**相似度**（通常用余弦相似度 Cosine Similarity 或点积 Dot Product）。
3. 返回相似度最高的 Top-K 个文档片段

BGE：将文本映射到固定维度的向量表示，使用Bi-Encoder网络，开源的中文向量编码器，支持预训练和微调训练，自定义程度高，支持垂直场景应用，**适合语义级召回。**

**3.混合检索：（最佳实践）**

1. **并行执行**：同时发起向量检索（Dense）和关键词检索（Sparse/BM25）。
2. **结果合并**：将两路召回的结果合并。

##### **一.提示词优化**

**设计和优化提示词，提升RAG系统的生成质量，安全性和任务适配性，使其可靠且符合实际需求。**

**自定义提示词**

领域特定：确保生成内容符合专业要求

任务导向：针对不同任务设计专用提示词，提高任务完成度

**安全提示词**

内容过滤：加入过滤提示词，防止生成有害或偏见内容。伦理约束：确保生成内容符合伦理标准

**核心能力提示词**

知识参考：优化提示词根据检索文档生成准确的内容。

逻辑表达提示词：设计提示词增强模型的逻辑推理能力，生成更连贯的回复。



##### **二.参数优化**

**对模型生成的相关参数根据实际情况设置**

**Stream:** 是否以流式传输的方式返回响应，适用于逐步显示结果的场景。

**temperature:**控制生成文本的随机性。值越高，结果越随机发散，值越低，结果越确定和严谨。

**max_tokens**: 限制生成文本的最大长度（以token为单位）。

**top_p**: 控制生成文本的多样性，通过设置top_p，模型会从概率质量最高的token中进行采样，从而可能提高生成内容的相关性。

## AI Native重构存量产品

### 第一性原理

基本原理思考法，指从最基本的、不可再分的真理或假设出发，通过逻辑推理来构建理解和解决问题。

核心思想是逆向工程，将一个问题拆解成最基本的构成部分，然后构建一个全新的解决方案。



### AI Native

是一套以通用人工智能AGI为核心，重构数字化系统的方法论和技术体系。

AI表示数字化系统充分利用AGI和适配AGI，而非传统AI来适应人类设计

Native表示数字化系统从设计之初即考虑AGI的能力，基于AGI原生并面向目标设计智能系统



### 人机协同的3种模式

#### 智能体概念

具有通用人工智能AGI能力的软硬件载体

**智能体：被定义为运行在特定环境下具有智能行为的计算机系统。具有感知，推理，决策和行动能力，并可通过学习和适应来改进性能，按照已有知识或自主学习，与其他智能体进行沟通协作，可自主完成设定的目标。**

也可概括：承载智能的主要实体，具有软件实体（对话机器人），物理实体（机器人/自动驾驶），或者两者结合的实体。



智能体分类：目标型，自主型，反应型，具身型，学习型

智能体特性：协作性，自主性，主动性，适应性，反应性

智能体能力：视觉/听觉/触觉，认知思考，学习迭代，预测分析，记忆存储，执行操作

##### Agentic Workflow

**Agentic Workflow（智能体工作流）** 是大模型应用开发中的一种先进范式。它标志着我们从“向 AI 提问并等待回答”的**对话模式**，转向了“让 AI 自主规划、使用工具、执行任务并自我修正”的**代理模式**。

- **传统 LLM 应用**：用户输入 -> 模型生成文本 -> 结束。（模型是**被动**的回答者）
- **Agentic Workflow**：用户给目标 -> 模型**思考**拆解任务 -> **调用工具**（搜索、代码、API）-> **观察结果** -> **反思/修正** -> 输出最终结果。（模型是**主动**的执行者）

**CoT vs. Agentic Workflow**

| 特性     | 思维链 (Chain of Thought, CoT)                               | 智能体工作流 (Agentic Workflow)                              |
| -------- | ------------------------------------------------------------ | ------------------------------------------------------------ |
| 本质     | 一次性推理。模型在内部生成一连串思考步骤，然后直接输出最终答案。 | 迭代式执行。模型在一个循环中运行：思考 -> 行动 -> 观察 -> 再思考。 |
| 工具使用 | 通常不使用外部工具，或仅在一次生成中模拟工具调用。           | 深度集成工具。可以真正执行代码、调用 API、搜索网络、读取文件。 |
| 纠错能力 | 一旦生成错误，无法回头修改（除非重新生成整个序列）。         | 自我修正。如果行动结果不符合预期，模型可以调整策略重试。     |
| 控制权   | 模型“黑盒”运行，人类难以干预中间过程。                       | 流程可监控、可干预，人类可以在关键节点审核或提供反馈。       |
| 适用场景 | 逻辑推理、数学题、创意写作。                                 | 复杂任务规划、数据分析、自动化运维、跨系统操作。             |



##### 常见的智能体工作流

**A. 单智能体循环 (Single Agent Loop)**

- **描述**：一个强大的 LLM 负责所有事情：规划、工具调用、反思。
- **流程**：`Goal -> Plan -> Act -> Observe -> Reflect -> Repeat -> Final Answer`
- **优点**：设置简单，灵活性高。
- **缺点**：对于极复杂任务，单个模型容易迷失方向或陷入死循环。
- **代表框架**：LangChain Agents, AutoGen (单代理模式)。

**B. 多智能体协作 (Multi-Agent Collaboration)**

- **描述**：将任务分配给多个具有不同“人设”或“专长”的智能体，它们互相沟通协作。

- 角色示例

  ：

  - **产品经理 Agent**：负责拆解需求。
  - **工程师 Agent**：负责写代码。
  - **测试工程师 Agent**：负责运行代码并找 Bug。
  - **审核员 Agent**：负责最终检查。

- **流程**：Agent A 输出 -> Agent B 接收并处理 -> Agent C 评审 -> 反馈给 A 修改。

- **优点**：模拟真实团队分工，解决复杂问题的能力极强，减少幻觉。

- **代表框架**：Microsoft AutoGen, CrewAI, LangGraph。

**C. 人机协同 (Human-in-the-Loop)**

- **描述**：在关键决策点（如执行删除操作、花费资金、发布代码）暂停工作流，请求人类确认。
- **价值**：确保安全性，防止智能体“失控”。

assistant层：适合信息获取需求，低风险容忍度，短期价值窗口

3.在低风险领域尝试agent自动化，收集数据

### 行业大模型的定义

**行业大模型**（也常称**垂直大模型 / 领域大模型**），是**基于通用大模型、针对特定行业深度定制**的专用大型人工智能模型，核心是用行业专属数据与知识做二次训练 / 微调，以解决专业场景问题。

一、核心定义

行业大模型是在**通用大模型**基础上，融合**行业专有数据、专业知识、业务规则与流程**，通过**继续预训练、指令微调、知识蒸馏**等方式优化，形成的**领域专精型大模型**。

- 定位：从 “通才” 转向 “专才”，聚焦单一行业（如医疗、金融、法律、工业、政务）。
- 本质：不是从零训练，而是**通用能力 + 行业深度**的结合。
- 目标：在专业任务上**准确率更高、幻觉更少、更符合行业规范**。

二、核心特征

1. 数据专属化

   - 训练数据：通用语料 + 行业文献、报告、案例、内部业务数据、专业知识库。
   - 强调：高质量、合规、结构化的**行业垂直数据**。

   

2. 能力专业化

   - 深度理解行业术语、规范、逻辑与流程。
   - 支持复杂专业任务：如医疗辅助诊断、金融风控、法律合同审查、工业故障诊断。

   

3. 场景深度化

   - 深度嵌入业务流程，从 “问答工具” 升级为**决策中枢**。
   - 适配行业合规、安全、隐私要求。

   

4. 效果精准化

   - 专业领域输出更可靠、可解释、可追溯。
   - 减少通用模型在专业场景的 “幻觉” 与错误。

   

三、与通用大模型的关键区别

| 维度     | 通用大模型（如 GPT-4、文心一言） | 行业大模型（如 Med-PaLM、BloombergGPT） |
| -------- | -------------------------------- | --------------------------------------- |
| **定位** | 跨领域通用能力，“一知万解”       | 垂直领域专精，“一行一解”                |
| **数据** | 全网海量通用文本                 | 通用数据 + 行业专属高质量数据           |
| **能力** | 泛化能力强，专业深度不足         | 专业深度强，跨领域能力弱                |
| **目标** | 通用问答、创作、多模态           | 解决行业痛点、支撑业务决策              |
| **成本** | 训练 / 推理成本极高              | 基于通用模型微调，成本更低、落地更快    |

四、典型代表与应用

- **医疗**：Med-PaLM、华佗大模型（辅助诊断、病历生成、医学文献解读）。
- **金融**：BloombergGPT（金融分析、风控、投研、合规审查）。
- **法律**：LawGPT（合同审查、案例检索、法规解读）。
- **工业**：工业大模型（生产优化、故障预测、质量检测）。
- **政务**：政务大模型（政策解读、智能审批、便民服务）。

五、一句话总结

行业大模型 = **通用大模型的基础能力 + 行业数据与知识的深度注入 + 业务场景的定制优化**，是 AI 从 “通用” 走向 “产业落地” 的核心形态。

解决的问题：

1. 解决「不懂专业术语」的问题

- 通用大模型：只懂日常语言，不懂行业黑话、标准、规范。
- 行业大模型：懂专业术语、懂流程、懂规范，能精准理解专业需求。

2. 解决「胡说八道（幻觉）」的问题

- 通用大模型：遇到专业问题容易瞎编、出错、不严谨。
- 行业大模型：用真实行业数据、知识库训练，输出更可靠、可追溯、少幻觉。

3. 解决「不符合业务流程」的问题

- 通用大模型：只会问答，不会干活，不能嵌入系统。
- 行业大模型：能对接业务流程，直接用于审批、诊断、风控、生产、运维等真实工作。

4. 解决「安全合规不可控」的问题

- 通用大模型：数据在外、逻辑黑盒，不敢用在敏感行业。
- 行业大模型：可私有化部署、数据不出域、符合监管，企业敢用、能用、放心用。

### 鲁棒性（稳健性）

**鲁棒性 = 模型在乱七八糟、不标准、有干扰的情况下，依然能正常工作的能力。**

**1）文字输入乱一点**

- 正常：“帮我写个工作总结”
- 有点乱：“帮我写个工作总结～谢谢啦！”
- 有错字：“帮我写个工做总洁”
- 语序怪：“工作总结帮我写一个”

**鲁棒性好**：都能看懂，正常回答。

**鲁棒性差**：稍微改一改就答非所问、乱输出。

------

**2）有干扰、有噪音**

- 语音识别有背景杂音
- 图片有点模糊、有遮挡
- prompt 里混了无关话

**鲁棒性好**：自动过滤噪音，抓住重点。



#### 解决方案

**1. 数据层面（最有效）**

- 加入**噪声数据**：错别字、乱序、省略句、口语化
- 加入**OOD 分布外数据**：领域外、无关问题、恶意提问
- 数据**清洗 + 去噪**，减少脏数据带来的不稳定

**2. 训练 / 微调层面**

- **对抗训练**：故意给坏输入，让模型学会不崩
- **SFT + 偏好对齐**（RLAIF/RLHF）：让输出更稳定
- 少用极端样本，避免模型学 “奇怪规律”

**3. 架构与推理层面**

- 温度 `temperature` 调低 → 更稳定、更少幻觉

- 使用 **约束解码**：限制格式、关键词、逻辑

- 加 

  校验器 / 判别器

  ：

  

  检查输出是否合理、是否胡说、是否跑题

**4. 工程层面（最容易落地）**

- 鲁棒 Prompt 工程

  - 明确边界：不知道就说不知道
  - 禁止编造、禁止扩展
  - 固定输出结构

  

- 输入预处理

  - 纠错、去重、过滤干扰内容

  

- 输出后处理

  - 事实校验
  - 格式校验
  - 风险过滤





# 垂直领域大模型微调实战



### 通用模型问题

**话术官僚化**：通用模型倾向输出**平衡，安全，泛化的废话**。缺乏行业洞察。

**逻辑浅表化**：能识别行业术语，但在处理连续的，强业务约束的agent任务时，无法保持连贯性，**逻辑链易断裂**

**事实幻觉：**在专业领域，通用模型的幻觉是致命的。

GPT-4o处理医疗领域：

总结临床记录，幻觉率高达50%

例病例存在隐含错误时，会**放大错误**并虚构治疗方案





### 为什么不能靠提示词工程解决问题

**一。内存焦虑**

1. 上下文限制：专业领域的SOP可能上万字，全部放到prompt可能**导致模型顾头不顾尾**

   研究表明答案位于文档头部时准确率超80%，文档中部仅40%-50%，还不如无文档回答

**二。稳定性陷阱**

Prompt是**软约束**，在agent复杂的”思考-行动-观察“循环中，某步未严格遵循指令，则这个任务崩盘。（多步任务中，单点失误会被放大）

**三。成本和效率**

长prompt = 高成本 ， token费用和推理延迟无法支持大规模业务





### 微调的本质：重塑模型的“思考惯性”

**微调在做什么：**微调解决了通用模型“急于给出建议”而非“先进行合规性审查”的顽疾

1. **修正概率分布，**面对行业问题，token概率选择天然符合专业习惯
2. **内化思维模式，**行为克隆：专家解题思路（action plan）变为模型的基础本能，学会“遇到A，先查B，再做C”的算法级流程

### 微调策略（千问）

**根据 SOP（标准化执行流程） 的复杂度定策略**

| SOP 特性                                        | 推荐微调策略                             | 结合逻辑                                                     |
| ----------------------------------------------- | ---------------------------------------- | ------------------------------------------------------------ |
| 格式/风格强约束 (如：必须用某种公文格式、语气)  | LoRA / QLoRA (轻量级微调)                | 仅需调整少量参数即可让模型学会特定的语言风格和输出模板，成本低，保留基座模型的通用能力。 |
| 逻辑流程复杂 (如：多步推理、严格的操作顺序)     | 全量微调 (Full Fine-tuning) 或 高秩 LoRA | 复杂的 SOP 逻辑需要更深度的参数调整，让模型内化推理路径，避免在长链条中迷失。 |
| 知识密集型 SOP (如：基于最新内部产品手册的回答) | RAG + 微调 (检索增强生成)                | 注意：静态知识不建议仅靠微调。SOP 中的动态知识应通过 RAG 检索，微调仅用于教会模型“如何调用检索工具”以及“如何根据检索结果按 SOP 组装答案”。 |
| 安全/合规红线 (如：严禁回答某些敏感问题)        | DPO / RLHF (偏好优化)                    | 构建“遵循 SOP 的安全回复”vs“违反 SOP 的危险回复”的对决数据，通过人类反馈强化学习，让模型在价值观层面 align SOP。 |



### 什么时候需要微调

提示词工程：成本最低，优先级高

RAG： 私有数据，实时新闻，打破知识瓶颈，不需要训练模型去记住这些数据（适合外挂知识库，RAG）

微调：提示词成本过高，使用少样本示例（few-shot）都无法处理行业问题，需要微调，深度优化

**微调核心触发场景：**

**深度对齐思考路径**  

**严格的输出控制**

**减少token消耗**

**长尾场景性能提升**  针对RAG无法覆盖的冷门或复杂边缘案例，增强模型原生能力

### workflow

假设任务是 **“调研竞争对手最新产品并生成对比PPT”**：

- **传统Workflow**：人工搜索 -> 人工复制粘贴 -> 人工写总结 -> 人工做PPT。

- **Agentic Workflow**：
1. **规划Agent**：拆解任务为“搜索信息”、“提取参数”、“撰写文案”、“制作幻灯片”。
  2. **搜索Agent**：调用搜索引擎，遍历竞品官网、新闻，抓取最新数据。
3. **分析Agent**：阅读抓取的内容，对比自家产品，生成结构化对比表格。
  4. **代码/工具Agent**：调用PPT生成库（如python-pptx），将表格和文案填入模板，生成文件。
  5. **反思环节**：如果发现某个竞品数据缺失，自动重新搜索或标记需人工补充。
  6. **输出**：直接交付一份初稿PPT给用户。

总结来说，**AI Agent Workflow** 是将大模型的推理能力与业务流程的执行能力深度结合的架构，它让AI从“被动问答”进化为“主动解决复杂问题”的生产力引擎。
